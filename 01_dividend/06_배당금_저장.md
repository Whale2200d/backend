# 배당금 프로젝트

## 자바 메모리 (선언한 Class를 사용할 때, 메모리에서는 어떤 일이 발생할까)
- 예시: Class에서 url을 멤버 변수(Member Variable)로 따로 분리해서 작성해서 사용함.
- 어떤 이점이 있을까
  - (코드 유지보수성 향상) Class 내 여러 곳에서 사용할 경우, 찾기 쉽고 수정이 용이함
  - (메모리 효율성 향상) 

- 자바 메모리
  - 생성자를 호출해서 인스턴스를 초기화(`scraper = new Scraper()`)하면, 해당 인스턴스(`scraper`)는 프로그램이 사용할 수 있는 메모리 공간 중 Heap의 일부를 할당받는다.
  - 클래스에 멤버 변수로 만든 값은 Heap에 저장이 된다.
  - 그런데, 클래스 내부에 정의해준 함수들에서 사용하는 변수(`scraper.scrap()`)는 메모리 공간 중 Stack에 생성된다.
  - 함수가 호출될 때, Stack 영역에 할당되고, 함수가 종료될 때 해제된다.
  - 같은 메서드를 동시에 호출하더라도, Stack에 각각 다른 영역을 할당받게 되고, 서로 간의 데이터는 공유되지 않는다.
  - scrap 함수에서 url을 선언했다면, 메서드를 여러 번 호출할 때마다 Stack에 url이 새롭게 쌓인다. url을 사용하는 만큼 메모리 공간을 할당한다. (비효율적)
  - 반면, 멤버 변수로 url을 사용하면, url은 Heap 영역에 할당되면서 scrap 메서드가 호출될 때마다, url을 참조해서 사용할 수 있다. (효율적)

- Q. 함수에 있는 모든 데이터는 전역 변수로 분리해서 사용하면 메모리 공간을 더 아낄 수 있지 않나?
  - Heap 메모리 영역은 한정되어 있으므로, 영역을 초과하게 되면 에러가 발생하면서 서비스가 죽어버리게 됨.

- Q. 자바 프로그램이 동작하게 되면 자연스럽게 Heap 공간에 데이터가 점점 쌓일 수 밖에 없는데?
  - 그래서 Java 내부에서는 주기적으로 사용되지 않는 인스턴스를 비워주는 Garbage Collection이라는 작업을 수행한다.

## Garbage Collection
- 메모리 할당 후 헤재가 안되면 메모리 누수 발생
- 수동으로 개발자가 직접 해제할 경우(C언어), 올바르게 해제되지 않아 버그 발생 가능성이 높음.
- 가비지 컬렉터는 동적으로 할당된 메모리 영역 중 더 이상 쓰이지 않는 영역을 찾아내서 해제
- 깊게 학습이 필요할 때, 어떤 메커니즘으로 동작하는지 등을 이해할 필요가 있다.
